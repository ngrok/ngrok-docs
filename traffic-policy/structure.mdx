A Traffic Policy document is a YAML or JSON object you can use to configure how traffic will be managed before reaching the apps and services at your ngrok endpoints. This reference guide outlines the schema for Traffic Policy documents.

## How Traffic Policy documents work

Traffic Policy Documents define how traffic is managed at different phases of the request. See [the phases documentation](/traffic-policy/concepts/phases) for a list of all available options.
Each phase corresponds to an event in the lifecycle of a request, such as when a connection is made, or when a response is sent.
In your Traffic Policy document, you can use these phases to define a list of [rules](/traffic-policy/concepts/phase-rules), which use [expressions](/traffic-policy/concepts/expressions) to filter traffic and [actions](/traffic-policy/concepts/actions) to manage traffic.

## Basic example

<CodeGroup>
```yaml policy.yml
# Phase names are the top-level keys.
on_http_request: # runs when a TCP connection is established
  - name: "Custom response for specific client IP"     # optional — a human-readable label for the rule
    expressions:  # optional — all conditions must be true for the rule to run
      - conn.client_ip == '1.2.3.4' # Any valid CEL expression. This one checks for a specific client IP
    actions:  # required — one or more actions to execute in order
      - type: custom-response
        config: # action-specific configuration; required fields depend on the action type
          status_code: 503
          body: <html><body><h1>Service Unavailable</h1><p>Our servers are currently down for maintenance. Please check back later.</p></body></html>
          headers:
            content-type: text/html
```
```json policy.json
{
  "on_http_request": [
    {
      "name": "Custom response for specific client IP",
      "expressions": [
        "conn.client_ip == '1.2.3.4'"
      ],
      "actions": [
        {
          "type": "custom-response",
          "config": {
            "status_code": 503,
            "body": "<html><body><h1>Service Unavailable</h1><p>Our servers are currently down for maintenance. Please check back later.</p></body></html>",
            "headers": {
              "content-type": "text/html"
            }
          }
        }
      ]
    }
  ]
}
```
</CodeGroup>

## Example with multiple rules and phases

The following example shows a complete policy that uses two phases and multiple rules:

<CodeGroup>
```yaml policy.yml
on_http_request:
  # Rule 1: block a specific IP address
  - name: "block bad actor"
    expressions:
      - "conn.client_ip == '1.2.3.4'"
    actions:
      - type: deny
        config:
          status_code: 403

  # Rule 2: require authentication on /admin paths
  - name: "admin oauth"
    expressions:
      - "req.url.path.startsWith('/admin')"
    actions:
      - type: oauth
        config:
          provider: google

  # Rule 3: no expressions—forward all remaining traffic to the upstream service
  - actions:
      - type: forward-internal
        config:
          url: https://example.internal

on_http_response:
  # Rule 1: add a security header to every response
  - actions:
      - type: add-headers
        config:
          headers:
            x-frame-options: "DENY"
```

```json policy.json
{
  "on_http_request": [
    {
      "name": "block bad actor",
      "expressions": ["conn.client_ip == '1.2.3.4'"],
      "actions": [
        { "type": "deny", "config": { "status_code": 403 } }
      ]
    },
    {
      "name": "admin oauth",
      "expressions": ["req.url.path.startsWith('/admin')"],
      "actions": [
        { "type": "oauth", "config": { "provider": "google" } }
      ]
    },
    {
      "actions": [
        { "type": "forward-internal", "config": { "url": "https://example.internal" } }
      ]
    }
  ],
  "on_http_response": [
    {
      "actions": [
        { "type": "add-headers", "config": { "headers": { "x-frame-options": "DENY" } } }
      ]
    }
  ]
}
```
</CodeGroup>
