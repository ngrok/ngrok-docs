Traffic Policy uses [CEL expressions](/traffic-policy/concepts/expressions/) and [interpolation](/traffic-policy/concepts/cel-interpolation) to dynamically route requests to different internal endpoints.
Route based on virtually any request attributeâ€”subdomain, path, headers, cookies, query parameters, geographic location, IP intelligence, and more.

<Note>
All routing examples use the [`forward-internal` action](/traffic-policy/actions/forward-internal/) to send traffic to internal endpoints.
See the action reference for configuration details.
</Note>

## Routing methods

| Method | Use case |
| --- | --- |
| [By subdomain](#by-subdomain) | Multi-tenant apps, customer-specific routing |

| [By time of day](#by-time-of-day) | Scheduled maintenance, traffic shifting |

## By subdomain

Route requests from `https://*.example.com` to corresponding internal services.

### Extract subdomain dynamically

<CodeGroup>

```yaml policy.yml
on_http_request:
- name: Route by subdomain
  actions:
  - type: forward-internal
    config:
      url: https://${req.host.split('.example.com')[0]}.internal
```

```json policy.json
{
  "on_http_request": [
    {
      "name": "Route by subdomain",
      "actions": [
        {
          "type": "forward-internal",
          "config": {
            "url": "https://${req.host.split('.example.com')[0]}.internal"
          }
        }
      ]
    }
  ]
}
```

</CodeGroup>

### Route specific subdomains

<CodeGroup>

```yaml policy.yml
on_http_request:
- name: Route API subdomain
  expressions:
  - req.host.startsWith('api.')
  actions:
  - type: forward-internal
    config:
      url: https://api-service.internal
- name: Route admin subdomain
  expressions:
  - req.host.startsWith('admin.')
  actions:
  - type: forward-internal
    config:
      url: https://admin-service.internal
- name: Route everything else to main app
  actions:
  - type: forward-internal
    config:
      url: https://main-app.internal
```

```json policy.json
{
  "on_http_request": [
    {
      "name": "Route API subdomain",

        "req.host.startsWith('api.')"

          "type": "forward-internal",

            "url": "https://api-service.internal"

        }
```

## By path

Route requests to different services based on URL path patterns.

### API versioning

<CodeGroup>

```yaml policy.yml

  expressions:
  - req.url.path.startsWith('/api/v1')
  actions:
  - type: forward-internal
    config:
      url: https://api-v1.internal
- name: Route v2 API
  expressions:
  - req.url.path.startsWith('/api/v2')
  actions:
  - type: forward-internal
    config:
      url: https://api-v2.internal
- name: Default to latest version
  expressions:
  - req.url.path.startsWith('/api')

  - type: forward-internal

      url: https://api-v2.internal
```

## By time of day

Route traffic to different services based on time.
Use this for maintenance windows or traffic shifting.
See [time variables](/traffic-policy/variables/time/) for available options.

<CodeGroup>

```yaml policy.yml
on_http_request:
- name: Route to maintenance page during off-hours
  expressions:
  - time.now.hour() >= 2 && time.now.hour() < 6
  actions:
  - type: forward-internal
    config:
      url: https://maintenance-service.internal
- name: Normal routing
  actions:
  - type: forward-internal
    config:
```

## Combine multiple conditions

Create complex routing logic by combining multiple conditions.

### Route by path and method

<CodeGroup>

```yaml policy.yml
on_http_request:
- name: Route write operations to primary
  expressions:
  - req.url.path.startsWith('/api')
  - req.method in ['POST', 'PUT', 'DELETE', 'PATCH']
  actions:
  - type: forward-internal
    config:
      url: https://primary-database.internal
- name: Route read operations to replica
  expressions:
  - req.url.path.startsWith('/api')
  - req.method == 'GET'
  actions:
  - type: forward-internal
    config:
      url: https://read-replica.internal
```
