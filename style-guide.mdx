title: Style Guide
description: Style guide and best practices for ngrok documentation.
---

This style guide documents all Vale rules used in the ngrok documentation. 
Vale is a prose linter that helps maintain consistency and quality across documentation.

## Time format (AM/PM)

Use "AM" or "PM" preceded by a space. 
Do not use lowercase, periods, or omit the space.

**Correct:**
- 3:00 PM
- 9:30 AM

**Incorrect:**
- 3:00PM
- 9:30am
- 3:00 P.M.
- 9:30 a.m.

## Capitalize ngrok-specific terms

Always capitalize these ngrok-specific terms:

**Correct:**
- Traffic Policy
- Traffic Policies
- Kubernetes Operator
- Kubernetes Operators
- Traffic Inspector
- Traffic Inspectors
- Cloud Endpoint
- Cloud Endpoints
- Agent Endpoint
- Agent Endpoints

**Incorrect:**
- traffic policy
- traffic policies
- kubernetes operator
- cloud endpoint
- agent endpoint

## Date format

Use the format "Month Day, Year" (e.g., "July 31, 2016"). Do not use numeric formats or day-first formats.

**Correct:**
- July 31, 2016
- December 25, 2023

**Incorrect:**
- 7/31/2016
- 31/7/2016
- 7.31.2016
- 31 July 2016

## Description field required

Every document must include a `description` field in the frontmatter. 
The description should be approximately 150 characters long and end with a period.

**Correct:**
```yaml
---
title: Getting Started
description: Learn how to get started with ngrok and create your first tunnel.
---
```

**Incorrect:**
```yaml
---
title: Getting Started
---
```

## Ellipses

Don't use three periods (`...`) as an ellipsis. 
If you need to reference UI text that contains an ellipsis, use the single-character ellipsis (`…`).

**Correct:**
- The dialog shows **Continue…** when ready.

**Incorrect:**
- The dialog shows **Continue...** when ready.
- Wait for the process to complete...

## Em dash spacing

Don't put spaces before or after an em dash or en dash.

**Correct:**
- The feature is available—check the settings.
- Use ngrok—it's fast and secure.

**Incorrect:**
- The feature is available — check the settings.
- Use ngrok — it's fast and secure.

## Exclamation points

Don't use exclamation points in documentation. 
Keep the tone professional and calm.

**Correct:**
- This feature is now available.
- You can configure the settings.

**Incorrect:**
- This feature is now available!
- You can configure the settings!

## First-person pronouns

Avoid first-person pronouns (I, me, my, mine, we, our, us, let's). 
Speak directly to the reader using "you" instead.

**Correct:**
- You can configure the settings.
- Your application will receive the request.
- To get started, follow these steps.

**Incorrect:**
- I can configure the settings.
- We recommend using this approach.
- Let's configure the settings.
- Our application will receive the request.


## Gender-neutral pronouns

Don't use gender-specific pronouns like "he/she" or "s/he".
Use gender-neutral alternatives like "they" or "them".

**Correct:**
- The user can configure their settings.
- Each developer should update their code.
- They can access the dashboard.

**Incorrect:**
- The user can configure his/her settings.
- Each developer should update s/he code.
- (S)he can access the dashboard.

## Heading capitalization

Use sentence-style capitalization in headings. 
Capitalize the first word and proper nouns only.

**Correct:**
- Getting started with ngrok
- Configure your API key
- Using Traffic Policies
- Introduction to Kubernetes

**Incorrect:**
- Getting Started With Ngrok
- Configure Your API Key
- Using Traffic Policies (if not a proper noun)

**Note:** Many proper nouns and technical terms are exceptions, such as "API", "Kubernetes", "Traffic Policy", "ngrok", etc. See the full list in the Vale rules.

## Heading punctuation

Don't put periods at the end of headings.

**Correct:**
- Getting started
- Configuration options
- Troubleshooting

**Incorrect:**
- Getting started.
- Configuration options.
- Troubleshooting.

## Latin abbreviations

Replace Latin abbreviations with plain English.

**Correct:**
- For example, you can use this method.
- That is, the configuration must be updated.

**Incorrect:**
- e.g., you can use this method.
- i.e., the configuration must be updated.
- eg, you can use this method.
- ie, the configuration must be updated.

## Hyphens after -ly words

Don't use hyphens after words ending in "-ly".

**Correct:**
- The recently updated feature
- A quickly running process
- The newly created account

**Incorrect:**
- The recently-updated feature
- A quickly-running process
- The newly-created account

## Optional plurals

Don't use optional plurals in parentheses like "user(s)". Choose either singular or plural based on context.

**Correct:**
- The user can configure settings.
- Users can configure settings.
- Select the option you need.

**Incorrect:**
- The user(s) can configure settings.
- Select the option(s) you need.

## Oxford comma

Use the Oxford comma (serial comma) in lists of three or more items.

**Correct:**
- You can use ngrok, Cloudflare, or AWS.
- The options include red, blue, and green.
- Configure the API, database, and cache.

**Incorrect:**
- You can use ngrok, Cloudflare or AWS.
- The options include red, blue and green.
- Configure the API, database and cache.

## Periods with acronyms

Don't use periods with acronyms or initialisms.

**Correct:**
- API
- HTTP
- HTTPS
- DNS
- CLI

**Incorrect:**
- A.P.I.
- H.T.T.P.
- H.T.T.P.S.
- D.N.S.
- C.L.I.

## Sentence spacing

Use exactly one space after sentence-ending punctuation (period, question mark, exclamation point).

**Correct:**
- This is a sentence. This is another sentence.
- Did you configure it? The settings are ready.
- Complete the setup. Then test the connection.

**Incorrect:**
- This is a sentence.  This is another sentence. (two spaces)
- This is a sentence.This is another sentence. (no space)

## American spelling

Use American spelling instead of British spelling.

**Correct:**
- organized
- color
- center
- labor

**Incorrect:**
- organised
- colour
- centre
- labour
