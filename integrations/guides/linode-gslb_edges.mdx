This guide shows how to layer ngrok's [Global Load Balancer](/universal-gateway/global-load-balancer/) (GSLB) on top of a VM-based deployment on Linode.

ngrok's GSLB routes traffic to the nearest Point of Presence without requiring you to deploy infrastructure, provision IPs, or change DNS records. It works on top of Linode's NodeBalancer for additional resiliency.

## What you'll need

- An [ngrok account](https://ngrok.com/signup).
- A [Linode](https://login.linode.com/signup) account.
- Three Ubuntu 24.04 VMs in three globally distributed regions (e.g., Atlanta, Osaka, Frankfurt) with unique hostnames and [Docker](https://docs.docker.com/engine/install/) + [Docker Compose](https://docs.docker.com/compose/install/) installed on each.
- [`jq`](https://jqlang.org/) on your local workstation.

## 1. Reserve your ngrok domain

[Generate an API key](https://dashboard.ngrok.com/api-keys) in the ngrok dashboard and export it:

```bash
export NGROK_API_KEY=<YOUR-API-KEY>
```

Reserve a subdomain of an [ngrok-managed domain](/universal-gateway/domains/) (e.g., `YOUR_COMPANY-linode-gslb.ngrok.app`) via the [ngrok dashboard](https://dashboard.ngrok.com/domains) or the API:

```bash
curl \
  -X POST https://api.ngrok.com/reserved_domains \
  -H "Authorization: Bearer ${NGROK_API_KEY}" \
  -H "Content-Type: application/json" \
  -H "Ngrok-Version: 2" \
  -d '{"description":"Linode Load Balancing","domain":"'${NGROK_DOMAIN}'"}'
```

Export your domain:

```bash
export NGROK_DOMAIN=<YOUR-NGROK-DOMAIN>
```

<Tip>
To use a custom domain instead of an ngrok-managed one, [follow the custom domains guide](/universal-gateway/custom-domains/) and use your domain in the export above.
</Tip>

## 2. Create a Cloud Endpoint

Save this Traffic Policy to `policy.yaml`, replacing `dc-atlanta`, `dc-osaka`, and `dc-frankfurt` with the location names you'll use for your VMs:

```yaml
on_http_request:
  - actions:
      - type: set-vars
        config:
          vars:
            - backend_index: ${rand.int(0, 2)}
            - backend_url: ${["https://dc-atlanta.internal", "https://dc-osaka.internal", "https://dc-frankfurt.internal"][vars.backend_index]}
      - type: forward-internal
        config:
          url: ${vars.backend_url}
          on_error: continue
      - type: custom-response
        config:
          status_code: 503
          body: "No upstream available."
```

This policy picks one of three [internal endpoints](/universal-gateway/internal-endpoints/) at random per request. When a VM is offline, `on_error: continue` falls through to the 503 response instead of returning an error.

<Tip>
For weighted distribution, see the [weighted backend migration guide](/guides/migration/edges-to-endpoints#weighted-backends--not-yet-supported).
</Tip>

Create the Cloud Endpoint:

```bash
jq -n \
  --arg tp "$(cat policy.yaml)" \
  --arg url "https://${NGROK_DOMAIN}" \
  '{"type":"cloud","url":$url,"description":"Linode GSLB","traffic_policy":$tp}' | \
curl \
  -X POST https://api.ngrok.com/endpoints \
  -H "Authorization: Bearer ${NGROK_API_KEY}" \
  -H "Content-Type: application/json" \
  -H "Ngrok-Version: 2" \
  -d @-
```

Export the `id` from the response:

```bash
export ENDPOINT_ID=ep_...
```

## 3. Install the ngrok Agent and an example workload on each VM

Repeat the following on **each** VM:

Clone the [demo repository](https://github.com/joelhans/ngrok-vps-gslb-demo):

```bash
git clone https://github.com/joelhans/ngrok-vps-gslb-demo
cd ngrok-vps-gslb-demo
```

Edit `ngrok.yml` with your authtoken (from the [ngrok dashboard](https://dashboard.ngrok.com/get-started/your-authtoken)) and set `url` to the internal endpoint for that VM:

```yaml
version: 2
authtoken: <YOUR_NGROK_AUTHTOKEN>
log_level: debug
log: stdout
tunnels:
  vps-demo:
    addr: 5000
    url: https://dc-<LOCATION_NAME>.internal
```

Start the containerized deployment:

```bash
HOSTNAME=$(hostname -f) docker compose up -d
```

## 4. Test Global Server Load Balancing

Ping your API to see which backend responds:

```bash
curl ${NGROK_DOMAIN}/api
[{"id":"1cf26269-ce8b-4f16-91f1-fdf7bc6d9e80","dc":"atlanta"}]
```

Run a batch to observe load balancing across all three VMs:

```bash
for i in `seq 1 20`; do curl -X GET "${NGROK_DOMAIN}/api"; done
```

Visit the [Endpoints dashboard](https://dashboard.ngrok.com/endpoints) to confirm your tunnels are online.

## 5. (Optional) Extend the Traffic Policy

Add response headers with the request's country code by updating `policy.yaml`:

```yaml
on_http_request:
  - actions:
      - type: set-vars
        config:
          vars:
            - backend_index: ${rand.int(0, 2)}
            - backend_url: ${["https://dc-atlanta.internal", "https://dc-osaka.internal", "https://dc-frankfurt.internal"][vars.backend_index]}
      - type: forward-internal
        config:
          url: ${vars.backend_url}
          on_error: continue
      - type: custom-response
        config:
          status_code: 503
          body: "No upstream available."
on_http_response:
  - name: Add ngrok headers
    actions:
      - type: add-headers
        config:
          headers:
            country: ${.ngrok.geo.country_code}
            is-ngrok: "1"
```

Patch the endpoint:

```bash
jq -n \
  --arg tp "$(cat policy.yaml)" \
  '{"traffic_policy":$tp}' | \
curl \
  -X PATCH "https://api.ngrok.com/endpoints/${ENDPOINT_ID}" \
  -H "Authorization: Bearer ${NGROK_API_KEY}" \
  -H "Content-Type: application/json" \
  -H "Ngrok-Version: 2" \
  -d @-
```

## What's next?

- Add regions by appending internal endpoint URLs to the `set-vars` list and patching your Cloud Endpoint.
- Scale within a region by starting additional agents pointing to the same internal URL—ngrok [pools](/universal-gateway/endpoint-pooling/) them automatically.
- Use the [ngrok Kubernetes Operator](/k8s/) to mix VM- and Kubernetes-based deployments.
- Run A/B tests by adjusting the distribution logic in `set-vars`.
